You are an expert Rust backend engineer. We have an MCP service called **xapimcp** that runs on a headless Linux server (no desktop, no GUI, no browser). It exposes MCP tools that wrap only a few X endpoints:

- Bookmarks fetch: `GET /2/users/{id}/bookmarks`
- Bookmark delete: `DELETE /2/users/{id}/bookmarks/{tweet_id}`
- Conversation lookup (currently implemented via recent search): `GET /2/tweets/search/recent?query=conversation_id:{id}`

This MCP is consumed by **Alt-AI-Labs/xbookmark-puller**, which calls xapimcp over **Streamable HTTP** (`/mcp`) and is the *only* place that enforces X rate limiting/pacing. xapimcp should remain a thin, stateless wrapper over X HTTP *except* for what’s required to keep OAuth tokens valid.

## Critical constraints (headless production)
- Production has **no browser/GUI**, so **no interactive auth** can happen there.
- The one-time OAuth authorization must happen **once on a developer laptop**.
- After that, the headless server must run **forever** using a stored **refresh token**, automatically refreshing access tokens and persisting refresh-token rotation, with **no human intervention** and **no public domain**.

## Current repo reality you must align with
- Auth is **OAuth2 refresh token** only: `X_REFRESH_TOKEN` or `X_REFRESH_TOKEN_FILE`, plus `X_CLIENT_ID`/`X_CLIENT_SECRET` (or legacy `X_CONSUMER_KEY`/`X_CONSUMER_SECRET`). **Do not** use `X_BEARER_TOKEN`, `X_ACCESS_TOKEN`, or `X_ACCESS_TOKEN_SECRET` in xapimcp (app-only bearer and OAuth 1.0a user tokens are removed).
- xapimcp calls `get_valid_x_access_token().await` and uses that for `.bearer_auth(...)` on X API requests.
- `X_USER_ID` is documented as **numeric user id**; if omitted, xapimcp resolves it via `GET /2/users/me`. (Do not assume usernames will work for `/2/users/{id}` — it expects an id.)
- Do not log access tokens, refresh tokens, or token substrings in production logs.

## What you must implement (OAuth2 PKCE + offline refresh)

### 1) Dependencies
Update `Alt-AI-Labs/xapimcp/Cargo.toml` to add **OAuth2 + PKCE** support using the `oauth2` crate with reqwest integration.

- Add: `oauth2 = { version = "4", features = ["reqwest"] }`
- Do **not** downgrade existing deps. `axum` is already present (currently `0.8`); keep it compatible with the repo. Only add features if you truly need them.

### 2) New module: `src/x_oauth.rs`
Create `src/x_oauth.rs` containing:

- **`XOAuthClient`** struct holding at minimum: `client_id`, `client_secret`, `redirect_uri`.
- Correct X OAuth 2.0 endpoints:
  - Authorize: `https://x.com/i/oauth2/authorize`
  - Token: `https://api.x.com/2/oauth2/token`
- **Full PKCE (S256)** support.
- Scopes must include:
  - `tweet.read users.read bookmark.read bookmark.write offline.access`
- Implement refresh logic that handles refresh token rotation:
  - `async fn refresh_access_token(...) -> Result<(String, Option<String>), McpError>`
  - Return `(access_token, new_refresh_token_opt)`
  - When X returns a new refresh token, the caller must persist it.

At the top of `x_oauth.rs`, include a short README-style block that explains the operational workflow end-to-end (laptop authorization → copy refresh token to server → headless auto-refresh forever).

### 3) Config: add refresh token + “get valid access token”
Update `ServerConfig` (`src/config/mod.rs`) to support offline refresh:

- Add field: `x_refresh_token: Option<String>` loaded from an env var (pick a clear name, e.g. `X_REFRESH_TOKEN` or `X_OAUTH2_REFRESH_TOKEN`).
- Add `async fn get_valid_x_access_token(&self) -> Result<String, McpError>` which:
  - Errors clearly if no refresh token is configured:
    - “No refresh token found — run `authorize-x` on your laptop first and copy the token to the server”
  - Uses the stored refresh token to request a fresh access token.
  - Detects refresh-token rotation and persists the new refresh token.

Persist strategy requirement:
- The headless server must survive restarts without losing the rotated token.
- Use whatever config persistence model the repo already uses (env file, secrets store, etc.). If the repo does not have one yet, implement a minimal, explicit approach and document it (but keep secrets out of git).

### 4) One-time authorization path (laptop only): use Option A

#### Option A: one-time authorization binary
Create `src/bin/authorize-x.rs` that you run on your laptop:

- Run with `cargo run --bin authorize-x`.
- Starts a tiny local HTTP callback server on `http://127.0.0.1:8080/callback`.
- Prints the full authorization URL.
- Waits for the callback, exchanges `code` for tokens, then **prints the refresh token clearly**.
- Prints instructions: “Copy this refresh token to your headless server config as `X_REFRESH_TOKEN`”.

### 5) Switch X API calls to use `get_valid_x_access_token()`
Modify the xapimcp X client (`src/x/mod.rs`) so every X request uses:

- `.bearer_auth(config.get_valid_x_access_token().await?)`

This includes `get_my_bookmarks`, `delete_bookmark`, `get_replies_to_tweet`, and `fetch_authenticated_user_id` (or unify the error type, see below).

### 6) Error handling + consistency
- Use `rmcp::model::ErrorData` (`McpError`) and `McpError::internal_error(...)` for tool-facing errors.
- Today the repo mixes `anyhow::Result` and `McpError` depending on function; align this so auth failures surface as MCP internal errors with clear messages.

### 7) Logging requirements
- Do **not** log access tokens, refresh tokens, or token substrings.
- If you need correlation, log a **non-reversible hash** of the token (e.g. truncated SHA256) or just log “token_present=true”.
